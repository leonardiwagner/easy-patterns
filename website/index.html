Factory Pattern

When to use?

  • Class have constructor with multiple parameters
  • Class have multiple constructors
  • You want to control how classes are instantiated

What problems does it solves?

  • Centralizes in one place (Factory) how class can be instantiated
  • Avoids the code smell of using new keyword inside other classes.

Using Factory Pattern to create different kinds of robots

The class Robot represents a robot with name, speed, stamina and strength attributes.

public class Robot { private String name; private int speed; private int stamina; private int strength; protected Robot(String name, int speed, int stamina, int strength){ this.name = name; this.speed = speed; this.stamina = stamina; this.strength = strength; } public String getName() { return this.name; } public int getSpeed() { return this.speed; } public int getStamina() { return this.stamina; } public int getStrength() { return this.strength; } }

as we're going to create different types of robots based on different attributes values, we created an enum to name each robot type.

public enum RobotType { SPEEDSTER, EXPLORER, FIGHTER }

Right, now the factory class itself!

The RobotFactory class has the create(RobotType robotType) method that by given RobotType returns a instance of Robot

public class RobotFactory { public Robot create(RobotType robotType){ switch (robotType){ case SPEEDSTER: return new Robot("speedster robot", 100, 80, 20); case EXPLORER: return new Robot("explorer robot", 50, 100, 50); case FIGHTER: return new Robot("fighter robot", 20, 80, 100); default: return new Robot("robot", 50, 50, 50); } } }

Yes, we've finished, that's all, easy eh? Example how we can use our factory:

Robot speedsterRobot = robotFactory.create(RobotType.SPEEDSTER); Robot fighterRobot = robotFactory.create(RobotType.FIGHTER); if(fighterRobot.getStrength() > explorerRobot.getStrength()) { System.out.println(fighterRobot.getName() + " is stronger than " + explorerRobot.getName()); }

Benefits

  • It's better to read create(RobotType.SPEEDSTER) rather than new Robot("speedster robot", 100, 80, 20)
  • Avoids wrong instatiation of a class, such as new Robot("slow speedster robot", 0, 0, 0)
  • If an attribute of some robot type needs to be changed, we'll do it only on RobotFactory class, instead looking for every Robot instantiation in our entire code.

Decorator Pattern

When to use?

  • We want to add new functionalities to a class at runtime.
  • Class have multiple constructors
  • You want to control how classes are instantiated

What problems does it solves?

  • Centralizes in one place (Factory) how class can be instantiated
  • Avoids the code smell of using new keyword inside other classes.

Using Decorator to attach new functionalities to our robots

We have the same Robot class, but now it implements the RobotInterface interface

public class Robot implements RobotInterface { private String name; private int speed; private int stamina; private int strength; public Robot(String name, int speed, int stamina, int strength){ this.name = name; this.speed = speed; this.stamina = stamina; this.strength = strength; } public String getName() { return this.name; } public int getSpeed() { return this.speed; } public int getStamina() { return this.stamina; } public int getStrength() { return this.strength; } }

Why this interface?

We create the RobotInterface interface based on what Robot do. Now any class that we create that implements RobotInterface can be interchangeable with Robot, cool eh?

public interface RobotSpec { String getName(); int getSpeed(); int getStamina(); int getStrength(); }

Creating a decoration to Robot

We can attach things to our robots, such as extra batteries. To do it, our decorator class should implement RobotInterface too, and receive the original Robot by constructor and store it at private RobotInterface robot. Why? In that way we just create our ExtraBatteriesDecorator pretty much like Robot, implementing all methods and returning the original values, except those who we want to decorate. In the ExtraBatteriesDecorator we're adding + 50 of stamina of original robot received by constructor, and also adding a description in the getName() method

public class ExtraBatteriesDecorator implements RobotSpec { private RobotInterface robot; public ExtraBatteriesDecorator(RobotInterface robot){ this.robot = robot;} @Override public String getName() { return this.robot.getName() + ", with extra batteries"; } @Override public int getSpeed() { return this.robot.getSpeed(); } @Override public int getStamina() { return this.robot.getStamina() + 50; } @Override public int getStrength() { return this.robot.getStrength(); } }

Yes, we

RobotInterface robot = new Robot("just a robot", 50, 50, 50); RobotInterface robotWithExtraBatteries = new ExtraBatteriesDecorator(robot); if(robotWithExtraBatteries.getStamina() > robot.getStamina())) { System.out.println(robotWithExtraBatteries.getName() + " has more stamina due extra batteries!"); }

Not impressed yet? Now things get better, we can decorate a decorated object! Yo Dawg!:

RobotInterface robotWithExtraBatteriesWithTurboWithSteelArmor = new SteelArmorDecorator(new TurboDecorator(robotWithExtraBatteries)); if(robotWithExtraBatteriesWithTurboWithSteelArmor.getSpeed() > robot.getStamina())) { System.out.println(robotWithExtraBatteriesWithTurboWithSteelArmor.getName() + " has more speed due turbo!"); } if(robotWithExtraBatteriesWithTurboWithSteelArmor.getStrength() > robot.getStrength())) { System.out.println(robotWithExtraBatteriesWithTurboWithSteelArmor.getName() + " has more strength due armor!"); }

Benefits

  • It's better to read create(RobotType.SPEEDSTER) rather than new Robot("speedster robot", 100, 80, 20)
  • Avoids wrong instatiation of a class, such as new Robot("slow speedster robot", 0, 0, 0)
  • If an attribute of some robot type needs to be changed, we'll do it only on RobotFactory class, instead looking for every Robot instantiation in our entire code.
